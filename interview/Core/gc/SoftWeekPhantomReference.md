##  Strong Reference
В Java есть Strong Reference (сильная ссылка) — это то, что мы пишем каждый день: Object obj = new Object(). Пока есть сильная ссылка, GC никогда не удалит объект.  

Но в реальности нам часто нужны ситуации, где мы говорим JVM: "Слушай, этот объект мне полезен, но если тебе не хватит памяти — удаляй его без сожалений".

## SoftReference (Мягкая ссылка)

Объект будет жить в памяти до тех пор, пока JVM не почувствует нехватку памяти (threat of OutOfMemoryError). Если памяти достаточно — объект будет жить вечно.

Зачем нужна (Философия): Для создания кэшей. Мы хотим держать данные в RAM для быстрого доступа, но не хотим, чтобы приложение упало с OOM, если кэш разросся слишком сильно.

```java
SoftReference<MyHeavyObject> softRef = new SoftReference<>(new MyHeavyObject());
MyHeavyObject obj = softRef.get(); // Вернет объект, если GC его еще не съел
```

## WeakReference (Слабая ссылка)

Объект не препятствует сборке мусора. Как только на объект не осталось сильных ссылок, при ближайшем цикле GC он будет удален. Weak будет удалена при первом же запуске GC, независимо от того, сколько свободной памяти.

Зачем нужна (Философия): Для канонизации объектов, предотвращения утечек памяти в коллекциях и метаданных.

### **Продакшен-реальность и нюансы:**  
WeakHashMap. Это классика. Ключи в этой карте — слабые ссылки. Если на ключ нет сильных ссылок вне карты, запись удалится. Пример из жизни: кэширование метаданных классов или парсинг XML, где мы не хотим держать в памяти объекты, которые больше нигде не используются.

## PhantomReference (Фантомная ссылка)
Самая загадочная и часто непонятная. У нее есть одна особенность: метод get() всегда возвращает null.  

Post-mortem cleanup (Посмертная очистка). Нам нужно точно знать момент, когда объект был удален из памяти, чтобы освободить связанные с ним нативные (off-heap) ресурсы.

Как это работает:    
PhantomReference обязана создаваться вместе с ReferenceQueue.  
Вы выделяете нативную память (например, через DirectByteBuffer или JNI).
Создаете PhantomReference на Java-объект, который "отвечает" за эту память.
Когда GC удаляет Java-объект, он не удаляет саму ссылку, а кладет её в ReferenceQueue.
Отдельный фоновый поток читает очередь и видит: "Ага, Java-объект умер, значит, нужно вызвать free() для нативной памяти". 

### Продакшен-реальность и нюансы:  
**DirectByteBuffer**. В Java именно так (через Cleaner, который внутри использует PhantomReference) освобождается off-heap память, выделенная для NIO буферов.
**Современный Java (9+).** Писать PhantomReference руками сейчас не нужно. В Java 9 появился интерфейс java.lang.ref.Cleaner. Он делает ровно то же самое, но с гораздо более приятным и безопасным API. Если на собесе спросят про Phantom, обязательно упомяните Cleaner — это покажет, что вы следите за эволюцией языка.
**Отличие от finalize()**. finalize() мог "воскресить" объект (присвоив его сильной ссылке), что ломало работу GC и вызывало жуткие тормоза. Cleaner / PhantomReference не дают объект воскресить. Объект уже мертв, мы просто чистим "мусор" после него.

## Связь с GC (как GC обрабатывает эти ссылки)

1. В JVM есть специальный класс ReferenceProcessor.  
2. Во время фазы Marking (пометки достижимых объектов), GC находит все объекты, на которые есть только Soft/Weak/Phantom ссылки.  
3. GC обрабатывает их строго в определенном порядке:  
Сначала очищает Soft (если не хватает памяти).  
Затем очищает Weak.  
В самом конце обрабатывает Phantom (помещает их в ReferenceQueue, но не удаляет из памяти, пока мы их не заберем из очереди и не вызовем clear()).
4. Важный нюанс: Пока PhantomReference лежит в ReferenceQueue и мы её не вызвали clear(), объект фактически остается в памяти (в состоянии "phantom reachable"). Если мы забудем очистить очередь, мы получим утечку памяти!
