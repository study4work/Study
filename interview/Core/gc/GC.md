### Сборщик мусора (Garbage Collector, GC) — это встроенная в Java автоматическая система управления памятью, которая находит объекты, ставшие ненужными программе, и удаляет их, чтобы освободить место для новых данных. В отличие от языков вроде C++, разработчику в Java не нужно вручную очищать память, что защищает от утечек информации и критических ошибок.

Области памяти в Java и их видимость.
Вся память Java-приложения (JVM) делится на две ключевые зоны с абсолютно разной областью видимости: 

Стек (Stack Memory): Своя выделенная память для каждого отдельного потока. Когда поток вызывает метод, в стеке создается блок (фрейм). 
Там хранятся только локальные переменные-примитивы (int, boolean) и ссылки на объекты. Как только метод завершается, его фрейм в стеке мгновенно и автоматически стирается. 
GC здесь не работает.

Куча (Heap Memory): Единое гигантское хранилище, общее для всех потоков приложения. 
Независимо от того, какой поток создал объект через ключевое слово new, этот объект попадает в Кучу. Стек лишь держит «адрес-ниточку» к нему. Именно в Куче работает Сборщик Мусора.

Как Куча делится на области (Поколения)
Проектировщики Java заметили эмпирический факт: большинство создаваемых объектов живут очень недолго (например, временная строка внутри цикла). 
Это назвали слабой гипотезой о поколениях. На основе этого Кучу разделили на три зоны.

![gc.png](/src/main/resources/images/gc.png)

1. **Young Generation (Молодое поколение)** — место, куда попадают все свежесозданные объекты. Делится на три части:
   1. Eden (Эдем): Самый первый «роддом» для объектов.
   2. Survivor 0 и Survivor 1 (S0 / S1): Две небольшие буферные зоны для объектов, выживших после первых проверок.
2. **Old (Tenured) Generation (Старое поколение)** — сюда переводятся «долгожители», которые смогли пережить множество проверок в Молодом поколении. Обычно это кэши, конфигурации и тяжелые долгоживущие сервисы.
3. **Metaspace (Метапространство)** — строго говоря, лежит вне Кучи (в системной памяти). Хранит метаданные классов, методы и статические переменные. GC заглядывает сюда редко.

#### Как работает алгоритм сборки мусора

Главная задача GC — понять, живой объект или мёртвый. Java использует алгоритм Tracing (Трассировка). <br>
Представьте дерево. У него есть корни — GC Roots (это активные потоки, локальные переменные в стеке прямо сейчас, статические переменные). <br>
GC начинает идти от этих корней по цепочкам ссылок. <br>
Если до объекта от корня можно дойти по цепочке «ниточек»-ссылок — объект живой.Если объект «оторвался» и к нему нельзя добраться от корней — он признается мертвым, даже если мертвые объекты ссылаются друг на друга.

### Какие виды GC бывают и их преимущества

#### 1. Serial GC
Самый старый сборщик. <br>
Работает в один поток. Когда он включается, вся программа замирает, пока одно ядро процессора чистит память.<br>
Плюс: Потребляет очень мало оперативной памяти на собственные нужды.<br>
Минус: Долгие паузы.<br>
Где нужен: В микроконтейнерах, CLI-утилитах или на слабых машинах с 1 ядром.<br>

#### 2. Parallel GC
Работает по тем же алгоритмам, что и Serial, но выполняет очистку в несколько потоков параллельно. <br>
Плюс: Максимальная пропускная способность (Throughput). Программа успевает обработать терабайты данных за минимальное процессорное время. <br>
Минус: Паузы Stop-The-World все еще ощутимые и долгие, так как приложение на время сборки полностью «встает». <br>
Где нужен: Системы пакетной обработки данных, тяжелые расчеты, где важна скорость вычислений, а секундная задержка не критична.<br>

#### 4. ZGC и Shenandoah
Ультрасовременные сборщики с низкой задержкой (Low-Latency). <br>
Они выполняют практически всю грязную работу (перемещение объектов) одновременно с работой приложения, не останавливая его.<br>
Плюс: Паузы Stop-The-World составляют меньше миллисекунды даже на кучах размером в 16 Терабайт!<br>
Минус: Снижают общую пропускную способность процессора примерно на 5-15% (поскольку ядра процессора делятся между кодом программы и параллельной уборкой).<br>
Где нужен: Финансовые системы, биржи, высоко нагруженные чаты и игровые сервера, где задержка даже в 50мс недопустима.<br>

#### Утечка памяти в Java (Memory Leak)
 Хотя сборщик мусора (GC) автоматически удаляет ненужные объекты, он делает это только тогда, когда на объект нет ни одной ссылки от GC Roots. <br>
 Утечка памяти в Java — это ситуация, когда программа больше не использует объект, но на него всё ещё непреднамеренно висит «забытая» ссылка. GC считает такой объект живым и не может его удалить. <br>
 Со временем кэши, списки или слушатели забивают память, что приводит к критической ошибке java.lang.OutOfMemoryError: Java heap space.<br>
 Самые частые причины утечек:Забытые статические коллекции: Поля со значением static живут всё время, пока работает приложение. <br>
 Если добавлять элементы в static List или static Map и забывать их оттуда удалять, эти объекты останутся в памяти навсегда. <br>
 Неудаленные слушатели (Listeners / Observers): Если зарегистрировать объект как слушатель событий в долгоживущем сервисе, сервис будет держать ссылку на этот объект, не давая GC его уничтожить. <br>
 Незакрытые ресурсы: Открытые сетевые соединения, потоки ввода-вывода (FileInputStream) или соединения с базами данных удерживают память ОС, пока их принудительно не закроют через close(). <br>
 Неправильный equals() и hashCode(): Если использовать объект как ключ в HashMap, но не переопределить эти методы, вы не сможете найти и удалить этот элемент из мапы, и он застрянет там навсегда.Как обнаружить утечку памяти? <br>
 #### Мониторинг (Профилирование): 
Использование инструментов вроде VisualVM, JProfiler или встроенного в IntelliJ IDEA Java Flight Recorder (JFR). 
На графике кучи (Heap) утечка выглядит как «пила», у которой нижние точки (после Full GC) со временем становятся всё выше и выше.
 #### Анализ дампов памяти (Heap Dump): 
 С помощью утилиты Eclipse Memory Analyzer (MAT) можно сделать снимок памяти (дамп) в момент, когда она почти заполнилась. <br>
 MAT автоматически рассчитывает «Leak Suspects» (подозрения на утечку) и показывает, какой именно класс или массив удерживает больше всего мегабайт.
