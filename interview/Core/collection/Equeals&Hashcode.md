## Контракт Equals&Hashcode

**Если объекты равны по Equals они обязательно должны быть равны по Hashcode.  
Если объекты равны по Hashcode, это не значит что они равны по Equals.  
Если объекты не равны по Hashcode - они гарантированно разные.  
Если объекты не равны по Equals - их hashcode может быть как разным так и одинаковым.** 

**equals()** — отвечает за равенство значений (проверяет, являются ли объекты логически одинаковыми на основе их данных).  
По умолчанию метод equals() в Java сравнивает ссылки на объекты в памяти. Это значит, что он проверяет, указывают ли две переменные на один и тот же экземпляр объекта.

**Как правильно переопределять его при наследовании?**  
К сожалению, идеального решения для наследования с добавлением "значимых" полей не существует. Есть несколько подходов:  
1: Композиция вместо наследования  
2: Запретить наследование  
3: Абстрактный базовый класс

 ## instanceof vs getClass()
Правило Joshua Bloch:  
Если класс не final и вы допускаете наследование — используйте getClass().  
Если класс final или не для наследования — можно instanceof.

Требования к equals:  
**Рефлексивность - x.equals(x) == true  
Симметричность - x.equals(y) == y.equals(x)  
Транзитивность - x=y, y=z → x=z  
Consistency - Результат стабилен во времени  
Null handling - x.equals(null) == false**

## Equals/hashCode в Hibernate entities
Использовать только immutable business keys  
НЕ использовать @Id в equals/hashCode  
НЕ использовать mutable поля  
Если нет business key → не переопределять (использовать identity по умолчанию)

## Что такое System.identityHashCode()?
это hashCode по умолчанию (от Object) основан на адресе объекта в памяти
