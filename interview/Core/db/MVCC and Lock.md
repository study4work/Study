## Multi-Version Concurrency Control 

Идея:  
• Каждая транзакция видит СВОЙ снимок (snapshot) данных  
• Чтение НЕ блокирует запись  
• Запись НЕ блокирует чтение  
• Хранятся НЕСКОЛЬКО версий одной строки  

Недостатки:  
Высокая конкурентность (readers не ждут writers)  
Нет deadlock от чтения  
Простая реализация уровней изоляции  

MVCC даёт конкурентный доступ на чтение и запись  
Возможны lost update  
Нужен optimistic/pessimistic lock

НО! Нужны важные уточнения:  
MVCC защищает ЧТЕНИЕ от записи (readers не ждут writers)  
Запись ВСЁ РАВНО блокирует запись (row-level locks  
• Writers блокируют writers  
• UPDATE строки → неявная блокировка этой строки  
• Вторая транзакция, пытающаяся UPDATE ту же строки → ЖДЁТ первую (даже в READ_COMMITTED!))  
Lost update возникает в паттерне read-modify-write

## Optimistic Locking
Идея:  
• НЕ блокируем строку при чтении
• Проверяем конфликты при ЗАПИСИ  
• Используем version field (или timestamp)  

Как работает:  
1. Читаем entity + version  
2. Изменяем entity в памяти  
3. При save/update проверяем: version == текущий в БД?  
   • Да → обновляем, increment version  
   • Нет → OptimisticLockException (кто-то изменил!)

Преимущества:  
Нет блокировок → высокая конкурентность  
Подходит для read-heavy приложений  
Работает с MVCC  

Недостатки:  
При конфликте нужно повторять транзакцию  
Не подходит когда много ретраев

## Pessimistic Locking  
Идея:  
• Блокируем строку ПРИ ЧТЕНИИ  
• Другие транзакции ждут, пока блокировка не снята  
• Используем SELECT ... FOR UPDATE  

Как работает:  
1. SELECT ... FOR UPDATE → блокируем строку  
2. Другие транзакции, пытающиеся прочитать/изменить → ждут  
3. Делаем изменения  
4. COMMIT/ROLLBACK → разблокировка

Преимущества:  
Гарантирует отсутствие конфликтов  
Не нужно повторять транзакцию  
Подходит для write-heavy приложений
