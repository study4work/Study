## Основные виды индексов (По структуре данных)

### B-Tree / B+ Tree (Сбалансированное дерево)  
Устройство:  
 **Многоуровневое дерево**. В B+ Tree (применяется в большинстве СУБД, например, PostgreSQL, MySQL InnoDB) все данные или ссылки на строки хранятся строго в нижних узлах (листьях), а внутренние узлы содержат только ключи для маршрутизации. Листья связаны в двусвязный список.  
**Предназначение**: 
Универсальный индекс по умолчанию. Идеален для точечного поиска, поиска по диапазону (BETWEEN, >, <), сортировки (ORDER BY) и группировки.  
**Сложность**:
Поиск, вставка и удаление выполняются за логарифмическое время \(O(\log N)\).

### Hash (Хеш-индекс)

**Устройство**:  
Хеш-таблица, преобразующая индексируемое значение в хэш-код, который указывает на конкретную корзину (bucket) со ссылкой на строку.  
**Предназначение**:  
Сверхбыстрый точечный поиск на точное совпадение (операторы = или IN).
**Минусы**:   
Совершенно бесполезен для поиска по диапазону, сортировки или поиска по части строки (LIKE 'Абв%').  
**Сложность**: В среднем поиск занимает константное время \(O(1)\).

### Bitmap (Битовый индекс)  
**Устройство**:   
Массив битов (0 и 1) для каждого уникального значения колонки, где позиция бита соответствует номеру строки в таблице.  
**Предназначение**:  
Используется для колонок с низкой кардинальностью (мало уникальных значений: Пол, Статус заказа, Страна) в аналитических СУБД (OLAP). Позволяет СУБД мгновенно объединять условия через побитовые операции AND / OR.  
**Минусы**:  
Крайне плохо переносит частые обновления (блокирует сразу много строк)  


### По логике
**Составной** (Composite / Composite Index): Индекс по двум и более колонкам.  

**Покрывающий** (Covering Index): Это не отдельный вид, а ситуация, когда индекс содержит все поля, запрашиваемые в SELECT. СУБД вообще не обращается к самой таблице на диске, отдавая данные напрямую из индекса (Index Only Scan).

**Частичный** (Partial / Filtered): Индекс, построенный только по части строк, удовлетворяющих условию (например, WHERE status = 'active'). Экономит место на диске.

**Функциональный** Обычный индекс хранит значения конкретной колонки. Функциональный индекс хранит результат вычисления функции или выражения, примененного к одной или нескольким колонкам таблицы.

### По организации данных
**Кластеризованный** (Clustered): Определяет физический порядок хранения самих строк таблицы на диске. Листья B+ Tree содержат сами строки с данными. У таблицы может быть только один кластеризованный индекс (обычно это Primary Key).

**Некластеризованный** (Non-Clustered / Вторичный): Хранится отдельно от таблицы. Листья содержат индексируемое значение и указатель на физическую строку (в Postgres — это ctid, в MySQL InnoDB — значение кластеризованного ключа).
