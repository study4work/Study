JIT (Just-In-Time compilation) — это компиляция “на лету” во время работы программы.

Традиционная компиляция:
Java Source → Compiler → Bytecode (.class)
JIT-компиляция (во время выполнения):
Bytecode → JIT Compiler → Native Code

Как работает JIT: Пошаговый процесс

![jit.png](/src/main/resources/images/jit.png)

1. Интерпретация байт-кода<br>
   Когда JVM запускается:
   Сначала весь байт-код интерпретируется (выполняется по инструкциям)
   Это медленно, но не требует времени на компиляцию
   JVM собирает статистику выполнения

2. Обнаружение "горячих" методов (Hot Methods)

   JVM отслеживает:
   Методы, которые вызываются часто (hot methods)
   Циклы, которые выполняются много раз
   Критические пути выполнения (hot paths)
   Пороговые значения:
   Client Compiler (C1): метод вызывается ~1500 раз
   Server Compiler (C2): метод вызывается ~10000 раз

3. Многоуровневая компиляция (Tiered Compilation)
   Современная JVM использует 4 уровня компиляции:

   Уровень 0: 
   Интерпретация (без компиляции)↓

   Уровень 1: 
   C1 компилятор (быстрая компиляция, базовые оптимизации)↓
    
   Уровень 2: 
   C1 компилятор (средняя скорость, больше оптимизаций)↓
   
   Уровень 3: 
   C1 компилятор (медленнее, еще больше оптимизаций)↓
    
   Уровень 4: 
   C2 компилятор (максимальная оптимизация, агрессивная)

4. Процесс оптимизации

![jit_1.png](/src/main/resources/images/jit_1.png)

Основные оптимизации JIT:
a) Inlining (Встраивание методов)


```java
// До оптимизации:
public int calculate(int a, int b) {
    return add(a, b);  // вызов метода
}

public int add(int x, int y) {
    return x + y;
}

// После inlining:
public int calculate(int a, int b) {
    return a + b;  // метод встроен напрямую
}
```

b) Dead Code Elimination (Удаление мертвого кода)

```java
// До оптимизации:
int x = 10;
int y = 20;
int z = x + y;  // если z нигде не используется
System.out.println("Hello");

// После оптимизации:
System.out.println("Hello");  // лишние вычисления удалены
```

c) Loop Unrolling (Развертывание циклов)

```java
// До оптимизации:
for (int i = 0; i < 4; i++) {
    sum += array[i];
}

// После unrolling:
sum += array[0];
sum += array[1];
sum += array[2];
sum += array[3];
```

d) Constant Folding (Вычисление констант)

```java
// До оптимизации:
int x = 10 * 20 + 5;

// После оптимизации:
int x = 205;  // вычислено на этапе компиляции
```

JIT хранит скомпилированный нативный код в Code Cache:
Быстрый доступ к уже скомпилированным методам
Ограниченный размер (по умолчанию 240MB в современных JVM)
При переполнении JIT отключается

Иногда JIT должен откатить оптимизации:
Если предположения оказались неверными
Если класс был переопределен (hotswap)
Возврат к интерпретации или менее оптимизированному коду


Типичные вопросы на собеседовании<br>
Q1: Чем отличается JIT от обычного компилятора?<br>
Ответ:<br
Обычный компилятор (javac) компилирует до выполнения (AOT)<br>
JIT компилирует во время выполнения (JIT)<br>
JIT может использовать профилирование runtime для оптимизаций<br>
JIT создает код под конкретное железо<br>

Q2: Что такое "warm-up" период в Java?
Ответ:<br>
Это время, когда JVM:<br>
Собирает статистику выполнения
Компилирует горячие методы<br>
Оптимизирует код<br>
Производительность растет со временем<br>

Q3: Как JIT решает, какие методы компилировать?<br>
Ответ:<br>
Считает количество вызовов метода (invocation counter)<br>
Считает количество выполнений циклов (backedge counter)<br>
Использует эвристические пороги<br>
Приоритет отдается критическим путям выполнения<br>
medium.com<br>

Q4: Что произойдет, если Code Cache переполнится?<br>
Ответ:<br>
JIT перестанет компилировать новые методы<br>
JVM выведет предупреждение: CodeCache is full<br>
Производительность резко упадет<br>
Решение: увеличить -XX:ReservedCodeCacheSize<br>

Q5: Можно ли отключить JIT?<br>
Ответ:<br>
Да, флагом -Xint (interpret-only mode):<br>
Только интерпретация<br>
Быстрый старт<br>
Очень медленная работа (в 10-50 раз медленнее)<br>
